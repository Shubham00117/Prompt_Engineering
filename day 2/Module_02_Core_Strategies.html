1Ô∏è‚É£ Using Delimiters and Formatting

Core Idea: Delimiters are special character sequences that act as "borders" in your prompt, separating instructions from content and data.
Type Example Best For
XML Tags <data>...</data> Claude, nested data, strict boundaries.
Triple Quotes """Text""" Python code, multi-line text blocks.
Markdown # Header, - List GPT models, clear hierarchy
Backticks `code` or ```block``` Code snippets, SQL queries, technical terms.
Dashes --- Separating major sections (e.g., Instructions vs Data).
Brackets [Instruction] System headers, placeholders, specific directives.

üöÄ Master Example (Using All Delimiters):

[SYSTEM_ROLE] 
Act as a Senior Python Developer.

# Task
Refactor the legacy code below.

---


The user is moving from Flask to FastAPI.



"""
def index():
    return "Hello World"
"""


[OUTPUT_RULES]
1. Return only the `FastAPI` code.
2. Do not explain.

*Notice how [Brackets] define roles, Markdown (#) defines structure, --- separates sections, XML holds context, Triple Quotes hold the raw text, and Backticks highlight specific technical terms.

2Ô∏è‚É£ Separating Instructions from Content

Core Idea: Prevent "Prompt Injection" or confusion by explicitly labeling what the AI should DO versus what it should process.

‚ùå Mixed Together (Bad)

"Translate 'The instructions say to press the red button' to Spanish."

‚úÖ Clear Separation (Good)

"Translate the text in <source> tags to Spanish.
<source>The instructions say to press the red button</source>"

2Ô∏è‚É£b Prompt Design Theory

Core Idea: Prompting is software engineering. Apply design patterns to your instructions.
Concept Explanation
Instruction Hierarchy Primary Goal > Constraints > Format. Structure prompts so the "Main Mission" is never lost in the details.
Minimalism vs Redundancy Minimalism: Less confusion, lower cost (Models like Claude prefer this).
Redundancy: Repeating key constraints at the end (Recency Bias) helps forgetful models (older GPTs).
Negative Prompting "Do NOT do X". Use sparingly. Models struggle with negation. Better to say "Keep sentences short" than "Do not write long sentences".

5Ô∏è‚É£-7Ô∏è‚É£ Zero, One, and Few-Shot Learning

Core Idea: "Shots" refer to the number of examples you provide to the LLM in the prompt. This is the fastest way to "teach" the model a new task.

5Ô∏è‚É£ Zero-Shot Prompting

Definition: Asking the model to perform a task without providing any examples. You rely entirely on the model's pre-trained knowledge.

Example (The Generic Ask):
Prompt: "Create a User ID for 'John Smith', born in 1990."
Response: "john_smith_1990" (or similar generic format).

Why it happens: Without guidance, the model guesses the most common format found on the internet.

6Ô∏è‚É£ One-Shot Prompting

Definition: Providing a single working example. This is often enough to set the output format or tone without needing a full list.

Example (The Format Hint):
Prompt: "Create a User ID. Example: 'Doe, John (1990)' -> 'DOE-1990'.
Now do it for 'Smith, John (1985)'."
Output: "SMITH-1985"

Why it works: One example bridges the gap between "Generic Guess" (Zero-Shot) and "Complex Logic" (Few-Shot). It's perfect for simple format copying.

7Ô∏è‚É£ Few-Shot Prompting

Definition: Providing a set of examples (usually 3-5) to demonstrate the exact pattern, tone, or format you want. This reduces ambiguity significantly.

Example (The Same Ask + Patterns):
Prompt: "Generate User IDs based on the examples:"
Input: "Alice Brown, 1985" -> Output: "BRO-A-85"
Input: "David Wilson, 2001" -> Output: "WIL-D-01"
Input: "John Smith, 1990" -> [AI Completes: "SMI-J-90"]

Why it works: The model ignores its generic training and applies the specific logic (First 3 letters of Last Name + Initial + Year) learned from your examples.
Technique Examples Given Best Use Case
Zero-Shot 0 General knowledge, summaries, creative writing.
One-Shot 1 Showing a specific output format (e.g., "Reply in JSON like this...").
Few-Shot 3-5 Complex logic, specific style mimicry, or edge cases.
üí° Key Insight: Pattern: Start with Zero-Shot. If it fails, try One-Shot (for format). If accuracy is still low, switch to Few-Shot (for logic).

7Ô∏è‚É£b Advanced Few-Shot Science

Core Idea: The quality and order of your examples matter more than the quantity.
üîÄ Formatting vs Reasoning
Examples teach Format easily (JSON, CSV).
Examples teach Reasoning poorly unless you use CoT in the example itself.
‚ö†Ô∏è Demonstration Leakage
If all your examples are about "Sports", the model might think the task is only about sports, even if you ask about Finance. Keep examples diverse!
üìâ Example Ordering
The last example has the most impact (Recency Bias). If the model consistently fails, try moving the hardest example to the end.
ü§ñ Synthetic Examples
Don't have data? Use a strong model (GPT-4) to generate 50 diverse examples, then use them as few-shot demonstrations for a cheaper model.
üí° Selection Strategy: K-Nearest Neighbors (KNN) For advanced systems, don't use static examples. Dynamically select the 3 examples from your database that are most similar (semantically) to the user's current question.

8Ô∏è‚É£ Chain-of-Thought (CoT) & Reasoning

Core Idea: Forcing the model to "show its work" before giving the final answer. This dramatically improves accuracy in math and logic. 
PROMPT: "I have 3 apples. I eat 1. I buy 5 more. I give 2 to my friend. How many do I have?"

WITHOUT CoT (Fast Thinking):
"You have 6 apples." (Often wrong due to rushing)

WITH CoT (Slow Thinking):
"Let's think step by step:
1. Start: 3 apples.
2. Eat 1: 3 - 1 = 2 apples.
3. Buy 5: 2 + 5 = 7 apples.
4. Give 2: 7 - 2 = 5 apples.
FINAL ANSWER: 5 apples."
Trigger Phrases:
‚Ä¢ "Let's think step by step."
‚Ä¢ "Walk through your reasoning process."
‚Ä¢ "Explain your math before the final result."
üí° Least-to-Most Prompting
For very hard logic: Break the problem into sub-problems. Ask the AI to solve the simplest part first, then use that answer to solve the next part. This "progressive solving" reduces errors in complex coding tasks.

9Ô∏è‚É£ Self-Verification & Error-Checking

Core Idea: Build a two-stage process where the model first generates, then audits its own output against your rules.

Stage 1: Generate

Prompt: "Write a SQL query to delete all users over 50 years old."
Output: "DELETE FROM Users WHERE Age > 50;"

Stage 2: Audit (The Guardrail)

Prompt: "Review the query above. Does it have a WHERE clause? Is it safe?"
Output: "Yes, WHERE clause exists. It is valid."

üîü Effective Example Crafting

Core Idea: The quality of your examples determines the quality of the output. Don't just give positive examples; teach the model what to do when things go wrong (Edge Cases).
‚ùå Bad Examples (Lazy)

Only showing "Happy Paths" confuses the model when real messy data arrives.

Input: "Email bob@test.com" 
Output: "bob@test.com"

Input: "Email sue@site.com" 
Output: "sue@site.com"
Result: If input is "Call 123-456", text model might lie and invent an email.
‚úÖ Good Examples (Robust)

Including "Edge Cases" teaches the model how to handle failure gracefully.

Input: "Email bob@test.com" 
Output: "bob@test.com"

Input: "Call 123-456" 
Output: "NO_EMAIL_FOUND"
Result: Model learns that "No Email" is a valid and expected answer.
üéØ The Golden Rules of Examples:
  • Diversity: Don't use 3 examples that look identical. Vary the length, style, and format.
  • Distribution: If your real data is 50% "Errors", your examples should be 50% "Errors".
  • Formatting: Always use clear separators like Input: and Output:.
